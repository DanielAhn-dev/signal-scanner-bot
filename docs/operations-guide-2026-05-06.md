# 웹 운영 기능 가이드

- 기준일: 2026-05-06
- 대상: signal-scanner-bot 웹 UI 운영자/사용자
- 범위: P0 포지션 관리 기능 + P1 운영 패널 (MVP)

---

## 1. 포지션 관리 (Portfolio 페이지)

### 1-1. 보유복구 (holdingrestore)

**언제 사용하나**
- 텔레그램 봇 없이 직접 보유 포지션을 복구해야 할 때
- 실제 매수한 종목이 가상 포트폴리오에 누락된 경우
- 기존 포지션의 매수가·수량을 완전히 초기화하고 싶을 때 (holdingedit와 다름)

**holdingedit와의 차이**

| 항목 | holdingedit | holdingrestore |
|------|-------------|----------------|
| 대상 | 기존 포지션만 수정 가능 | 없으면 신규 생성 |
| 종목 입력 | 코드 직접 지정 | 코드 입력 후 DB 존재 확인 |
| 거래 기록 | 기록 없음 | ADJUST 로그 기록 |
| 주용도 | 단가/수량 소폭 수정 | 누락 포지션 완전 복구 |

**사용법**
1. 포트폴리오 페이지 상단 "보유복구" 버튼 클릭
2. 종목코드 입력 (예: `005930`)
3. 수량 입력 (예: `5`)
4. 매수가 입력 (예: `75000`)
5. "실행" 클릭

**주의사항**
- 종목코드는 stocks 테이블에 등록된 코드만 허용 (미등록 코드 입력 시 404 오류)
- 기존 포지션이 있으면 매수가·수량을 덮어씁니다 (로트도 초기화)
- 거래 이력에 ADJUST 타입으로 기록됩니다

---

### 1-2. 보유수정 (holdingedit)

**언제 사용하나**
- 이미 보유 중인 종목의 단가나 수량을 소폭 수정할 때
- 포지션 자체는 있으나 수치만 잘못 기록된 경우

**사용법**
1. 포트폴리오 카드에서 해당 종목의 "보유수정" 버튼 클릭
2. 수량·매수가 수정 후 "실행"

---

### 1-3. 관심초기화 (watchreset)

**언제 사용하나**
- 보유수량이 0인 관심 종목을 일괄 정리할 때
- 포지션 목록이 너무 많아 지저분해진 경우

**사용법**
- 포트폴리오 상단 "관심초기화" 버튼 클릭 → 확인 모달에서 실행
- 보유 종목(quantity > 0)은 삭제하지 않음

---

### 1-4. 전체매도 (liquidateall)

**언제 사용하나**
- 모든 보유 포지션을 한 번에 청산할 때 (포트폴리오 리셋)

**동작**
1. 현재가(close) 기준으로 SELL 거래 기록 생성
2. 전체 포지션 quantity=0, status='interest'로 변경
3. 로트 잔량 0으로 마감

**주의사항**
- 되돌릴 수 없습니다. 거래 기록은 남지만 포지션은 초기화됩니다.

## 2. 운영 패널 (Operations 페이지)

메뉴: 운영 > 운영 패널

### 2-1. 자동사이클 (autocycle)

**동작 방식**
- 보유/관심 종목의 점수를 재평가하여 매수·매도 조건 충족 시 가상 거래를 실행합니다
- 내부적으로 `virtualAutoTrade` 작업을 jobs 큐에 등록합니다
- 봇 서버(cron worker)가 큐를 소비하면 실제 처리됩니다

**점검(dry-run) vs 실제 실행**
- **점검 실행**: 거래 없이 어떤 종목이 조건에 맞는지 시뮬레이션만 합니다 (권장)
- **실제 실행**: 조건 충족 종목에 대해 가상 매수/매도를 실행합니다

**주의사항**
- 작업이 즉시 실행되지 않고 큐에 적재됩니다. 봇 cron 주기에 따라 처리됩니다
- 중복 요청 방지를 위해 dedup_key를 사용합니다

---

### 2-2. 순차트리거 (autotrigger)

**동작 방식**
- 장중/장전 단계별 운영 워크플로우를 단계적으로 실행합니다
- 내부적으로 `virtualAutoTradeIntraday` 작업을 jobs 큐에 등록합니다

**단계 선택**
- `장중 (intraday)`: 장 중 시간대에 사용, 실시간 조건 재평가
- `장전 (ready)`: 장 시작 전 플랜 확인 및 진입 준비

**현재 MVP 상태**
- 현재는 dry-run 모드로만 요청합니다 (실제 실행은 향후 지원 예정)

---

### 2-3. 자동매도 점검 (autosellcheck)

**동작 방식**
- 현재 보유 중인 종목의 현재가 기준 수익률을 조회합니다
- 서버 API 호출 없이 웹에서 바로 계산 결과를 표시합니다

**결과 해석**
- 음수 수익률이 클수록 자동매도 조건에 해당할 가능성이 높습니다
- 실제 자동매도 조건(점수 기반)은 봇 내부 로직에서 결정되며, 이 뷰는 현황 파악용입니다

---

### 2-4. 최근 실행 이력

- 해당 chat_id 기준 최근 20건의 virtual_trades 기록을 표시합니다
- 매수(BUY), 매도(SELL), 조정(ADJUST) 구분
- memo 필드에 실행 소스(`web-restore:v1`, `웹 전체매도` 등)가 기록됩니다

## 3. 환경 변수 요약

| 변수명 | 설명 |
|--------|------|
| `UI_TRUSTED_WEB_ORIGINS` | CORS 허용 오리진 (콤마 구분) |

| `SUPABASE_SERVICE_ROLE_KEY` | Supabase 서비스 역할 키 |

## 4. 트러블슈팅

| 증상 | 확인 사항 |
|------|-----------|
| 보유복구 404 오류 | 종목코드가 stocks 테이블에 없음 — 코드 확인 |
| 운영 패널 401 오류 | chat_id 미연동 — 프로필 모달에서 텔레그램 연동 필요 |
| 자동사이클 요청 후 반응 없음 | 봇 cron worker 실행 중인지 확인, jobs 테이블 status 확인 |
| 전체매도 후 포지션 남음 | liquidateall은 quantity>0 기준 — 이미 0인 항목은 건너뜀 |
